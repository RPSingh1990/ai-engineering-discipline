# Recognition Plan

The goal is not to collect badges. The goal is to make this repository useful enough that AI builders, engineering leaders, and product leaders save it, star it, and reuse parts of it.

## Positioning

Use this angle:

> Vibe coding is useful for speed. It becomes dangerous without engineering discipline. This repo is a practical operating model for AI-assisted software teams.

Avoid:

- "ultimate framework"
- "replace engineers"
- "10x with prompts"
- generic agent hype

## GitHub Trust Targets

### First Target: 16 Stars

Why: unlocks the first meaningful public attention marker and proves the repo is useful beyond your own work.

Plan:

1. Publish the repo with a strong README.
2. Add one diagram or screenshot later.
3. Share one LinkedIn post with a concrete problem: "AI-built apps break after the demo."
4. Ask for feedback, not stars.
5. Improve based on issues and comments.

### Second Target: 5 External Issues Or PRs

Why: community feedback is a stronger signal than passive likes.

Good first issues:

- add a new agent eval
- improve the security scan
- add a CI example
- add a real failure mode
- add a public-safe agent template

### Third Target: 16 Merged PRs Across AI OSS

Why: public contribution history builds more trust than a profile badge.

Targets:

- documentation fixes in AI agent projects
- eval examples
- security notes
- sample apps
- CI templates

## LinkedIn Launch Sequence

### Post 1: Problem

Theme:

AI coding is not the problem. Unreviewed AI coding is the problem.

Call to action:

Ask engineering leaders what gate they wish AI-generated code always passed.

### Post 2: Build Log

Theme:

Show the repo structure and explain why it has templates, evals, and security checks.

Call to action:

Invite people to copy the PR checklist.

### Post 3: Field Lesson

Theme:

The most useful agent rule is not "write better prompts." It is "give each agent a job, permissions, and evals."

Call to action:

Ask for failure modes from other teams.

## Repository Improvements To Earn Stars

Add in this order:

1. One visual architecture diagram.
2. One working GitHub Action demo.
3. More agent examples: frontend, QA, code review, research.
4. A `validate-agent` command for agent specs.
5. A static demo report generated by the validator.
6. A short video or GIF.

## What Not To Do

- Do not publish raw private agent prompts.
- Do not turn this into a massive framework.
- Do not overclaim enterprise readiness.
- Do not chase novelty badges like YOLO.
- Do not ask people to star before the repo is useful.
